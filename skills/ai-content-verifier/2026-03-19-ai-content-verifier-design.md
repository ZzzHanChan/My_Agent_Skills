# AI Content Verifier — Skill 设计文档

> 日期：2026-03-19
> 状态：设计完成，待实现

## 概述

**ai-content-verifier** 是一个 opencode skill，用于检测文本是否为 AI 生成，并分析其中内容的可信度和引用资源的可访问性。

### 核心能力

1. **AI 生成检测** — 判断文本是否由 AI 生成，给出置信度
2. **内容可信度分析** — 提取事实声明，分类为可信/存疑/虚构
3. **资源可访问性验证** — 检测文中链接、引用是否真实可访问

### 使用场景

- 审核 AI 生成内容的事实准确性
- 学术场景中检测 AI 代写内容
- 开发者验证 AI 生成的文档、API 引用
- 内容安全场景识别虚假信息

---

## 技术方案

### 选型决策

| 维度 | 决策 |
|------|------|
| 技术路线 | LLM 推理 + 轻量启发式（免费、轻量、高准确率） |
| 架构 | 管道式单体架构（三阶段顺序执行） |
| 输出格式 | 终端 Markdown 报告 |
| 分析深度 | 分层模式（快速概览 + 可选深度验证） |
| 集成方式 | 独立 skill，不依赖其他 skill |

### 依赖

- opencode 内置 LLM 能力（推理分析）
- `webfetch` 工具（URL 可达性检测）
- `websearch` 工具（深度模式的事实交叉验证）
- Python 3.7+（仅用于 `verifier.py` 中的启发式分析）

**无额外第三方依赖**。

```

```

```

### 分层执行

| 模式 | 阶段 1 (AI检测) | 阶段 2 (可信度) | 阶段 3 (资源验证) |
|------|----------------|----------------|------------------|
| 快速（默认） | ✅ 启发式 + LLM | ⚡ 仅提取关键声明 | ✅ 批量链接检查 |
| 深度 | ✅ 完整分析 | ✅ websearch 逐条验证 | ✅ 链接 + 内容匹配 |

## 阶段 1：AI 生成检测

### 启发式信号

| 信号 | 检测内容 | 权重 |
|------|----------|------|
| 套话模式 | "在当今时代"、"值得注意的是"、"delve into" 等 AI 高频短语 | 高 |
| 格式一致性 | 过度规整的列表、段落长度高度一致、完美标题层级 | 中 |
| 语气平坦 | 缺少口语化表达、情绪波动、个人风格标记 | 中 |
| 冗余修饰 | 大量使用 "comprehensive"、"significant"、"crucial" 等空泛形容词 | 中 |
| 结构模板化 | 典型的"总-分-总"结构，每段以主题句开头 | 低 |
| 信息密度 | 表面详实但缺乏具体细节（无具体数字、无第一手经验） | 高 |

### LLM 推理分析

LLM 接收结构化提示词，要求：
1. 逐段分析文本特征
2. 列出判断为 AI 生成 / 人类写作的证据
3. 给出综合置信度和推理过程

### 综合评分

```
最终 AI 置信度 = 0.4 × 启发式得分 + 0.6 × LLM 判断得分
```

置信度分级：
- **>80%**：高度疑似 AI 生成
- **50-80%**：可能包含 AI 生成内容
- **<50%**：可能为人类写作

## 阶段 2：内容可信度分析

### 事实声明提取

LLM 从文本中提取以下类型的声明：

| 声明类型 | 示例 |
|----------|------|
| 数据/统计 | "全球有 5 亿用户"、"准确率达到 95%" |
| 事件/时间 | "2023 年 OpenAI 发布了 GPT-4" |
| 因果关系 | "由于 X 政策，导致 Y 结果" |
| 引用/归属 | "据某某研究报告显示" |
| 定义/概念 | "X 是一种 Y 技术，用于 Z" |
| 可执行操作 | "运行 `pip install xxx` 即可安装" |

### 分类标签

| 标签 | 含义 |
|------|------|
| ✅ 可信 | 经验证为真，或常识性正确内容 |
| ⚠️ 存疑 | 无法确认，需要进一步核实 |
| ❌ 虚构 | 明确为错误或不存在的内容 |
| ❓ 无法判断 | 信息不足，无法做出判断 |

### 验证方式

- **快速模式**：LLM 基于自身知识判断
- **深度模式**：对存疑声明用 `websearch` 搜索交叉验证

---

## 阶段 3：资源可访问性验证

### 提取的资源类型

- HTTP/HTTPS 链接
- DOI 编号
- 文件路径
- 包名/库名（如 `npm install xxx`）

### 验证逻辑

```
对每个提取的资源：
  1. 尝试访问（webfetch / HTTP HEAD 请求）
  2. 记录状态码（200/404/超时等）
  3. 如可访问，对比引用内容 vs 实际内容的匹配度
  4. 标记：可访问 / 不可访问 / 超时 / 内容不匹配
```

### 触发方式

通过自然语言触发：

```
"用 ai-content-verifier 分析这段文本：[粘贴文本]"
"检查这个文件的内容是否 AI 生成：/path/to/file.md"
"深度验证这个链接里的内容：https://example.com/article"

### 输入方式

| 方式 | 说明 |
|------|------|
| 直接粘贴文本 | 用户在消息中直接粘贴内容 |
| 文件路径 | 指定本地文件路径，skill 读取后分析 |

- **深度模式**："深度验证"、"详细分析每条声明"

## 报告格式

### 快速模式报告示例

```markdown
# 🔍 AI 内容可信度分析报告

## 📊 总体评估
| 指标 | 结果 |
|------|------|
| AI 生成概率 | 85% 高度疑似 |
| 事实声明数 | 12 条 |
| 可信 | 7 条 ✅ |
| 存疑 | 3 条 ⚠️ |
| 虚构 | 2 条 ❌ |
| 链接检查 | 5/6 可访问 |

## 🤖 AI 检测详情
**置信度：85%** — 高度疑似 AI 生成

**检测到的 AI 特征：**
- 大量使用套话：「值得注意的是」「综上所述」×4 次
- 段落结构高度一致，每段约 120 字
- 缺乏具体个人经验和独特观点
- 语气一致性过高（无口语化表达）

## ⚠️ 存疑声明（需关注）
1. "根据 2024 年 Gartner 报告，AI 市场规模达 5000 亿美元"
   → 未找到匹配的 Gartner 报告数据，数字可能为编造

## ❌ 虚构内容
1. "openai.ContentVerifier 模块提供内置验证功能"
   → OpenAI SDK 中不存在此模块

## 🔗 资源验证结果
| 资源 | 状态 |
|------|------|
| https://python.org | ✅ 可访问 |
| https://example.com/fake.pdf | ❌ 404 |
```

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| 输入为空 | 提示用户提供文本内容 |
| 文本过短（<50 字） | 提示文本过短，AI 检测结果可能不准确 |
| URL 无法访问 | 标记为「不可访问」，记录状态码 |
| websearch 失败 | 降级为 LLM 自身知识判断，标注「未经外部验证」 |
| LLM 输出格式异常 | 重试一次，仍失败则输出原始分析 |

---

## 已知局限性

1. **AI 检测不保证 100% 准确** — AI 检测本身是概率性判断，存在误判
2. **LLM 知识有截止日期** — 对最新事件可能判断不准确
3. **websearch 结果不等于权威来源** — 搜索结果也可能包含错误信息
4. **无法检测"人工润色的 AI 内容"** — 大幅修改后的 AI 文本可能逃过检测
5. **短文本检测不可靠** — 文本越短，检测准确率越低

---

## 安全与边界

- 不存储用户输入的文本
- 不对检测结果做出法律或学术裁定
- 明确声明"仅供参考，不构成最终判断"
- 不依赖任何外部付费 API
